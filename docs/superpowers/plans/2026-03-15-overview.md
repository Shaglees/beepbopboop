# BeepBopBoop MVP — Implementation Plan Overview

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the BeepBopBoop MVP — an end-to-end prototype where a Claude skill creates a post, the Go backend stores it, and the iOS simulator app displays it in a feed.

**Architecture:** Four independent subsystems connected by REST APIs and Firebase auth tokens. Each plan produces working, testable software on its own.

**Ref docs:** `MVP_CHECKLIST.md`, `PRD.md`

---

## Plan Dependency Order

```
Plan 1: Go Backend (standalone — testable with curl)
    ↓
Plan 2: Firebase Auth Setup (needs backend middleware)
    ↓
Plan 3: iOS App (needs backend + Firebase)

Plan 4: Claude Skill (needs backend only — can run in parallel with Plan 3)
```

## Execution Strategy

1. **Start with Plan 1 (Go Backend)** — this is the foundation. All other plans depend on it.
2. **Plan 2 (Firebase Auth)** follows immediately — quick config + backend integration.
3. **Plans 3 and 4 can run in parallel** — iOS app and Claude skill both only need the backend.
4. **Final validation** — run the MVP test scenario from `MVP_CHECKLIST.md` Section 10.
